# 场景定义：QA/测试

## 典型特征

- 添加或改进测试覆盖
- 修复 flaky tests
- 测试基础设施改进
- 通常不涉及业务逻辑变更

## 常见类型

1. **单元测试**
   - 覆盖率提升
   - 边界条件测试

2. **集成测试**
   - API 测试
   - 数据库集成测试

3. **E2E 测试**
   - 用户流程测试
   - 跨浏览器测试

4. **测试基础设施**
   - CI/CD 改进
   - 测试数据管理
   - Mock 服务

## 常见决策点

1. **测试策略**
   - 测试金字塔平衡
   - 重点覆盖区域

2. **测试数据**
   - 真实数据 vs Mock
   - 数据隔离方案

3. **执行策略**
   - 并行执行
   - 选择性执行

## Triage 策略

- **Level 判定**: 通常为 Level 2-3
  - 添加测试 → Level 2
  - 测试架构改进 → Level 3
- **关键信息**: 覆盖率目标、执行时间、CI 影响
- **常见风险**: 测试不稳定、执行时间过长

## 特殊考虑

QA 场景通常不需要复杂的审批流程，但需要关注：
- 是否影响 CI 执行时间
- 是否引入 flaky tests
- 覆盖率提升目标
